# Исследование: Telegram Mini Apps для записи клиентов

> Цель: понять рынок, найти лучшие практики и слабые места конкурентов
> Контекст: строим Mini App для записи на консультацию по ремонту бадминтонной ракетки

---

## 5 примеров приложений

---

### 1. YClients + Telegram (Referof Mini App)

**Что делает:** Крупнейшая в СНГ система управления записью для сфер красоты, медицины, фитнеса. Telegram Mini App через интеграцию с сервисом Referof — программа лояльности + запись прямо в мессенджере. Клиент открывает Mini App по ссылке/QR, видит доступных мастеров, выбирает услугу, дату и время.

**Сильные стороны:**
- Полная интеграция с CRM — записи мгновенно попадают в журнал мастера
- Выбор конкретного специалиста с профилем и фото
- Просмотр реального расписания: только свободные слоты
- Онлайн-оплата и предоплата прямо в Mini App
- Автоматические напоминания клиенту за 24ч и 1ч
- Программа лояльности (баллы, кэшбек)

**Слабые стороны:**
- Требует платной подписки на YClients (от 1800 ₽/мес) — недоступно для маленьких мастеров
- Интерфейс перегружен функциями, которые не нужны единственному мастеру
- Сложная настройка интеграции: нужен разработчик или агентство
- Mini App ощущается как сайт внутри Telegram, а не нативное приложение
- Нет обратной связи от клиента (например, описать проблему текстом)

**Что мы можем сделать лучше:**
- Простота: один мастер, один поток — без лишнего
- Поле для описания проблемы (что именно сломалось) — критично для нашей ниши
- Нативный дизайн под тему Telegram без "сайтовых" элементов
- Бесплатный старт, без абонентской платы

### 2. Calendry (Telegram Calendar Mini App)

**Что делает:** Аналог Calendly внутри Telegram. Позволяет создавать ссылку на расписание и давать её клиентам — те открывают Mini App, видят свободные слоты и бронируют встречу/консультацию. Поддерживает групповые чаты: можно найти общее удобное время для нескольких участников.

**Сильные стороны:**
- Минималистичный интерфейс: буквально 2 экрана
- Мгновенная авторизация через Telegram — пароли не нужны
- Синхронизация с Google Calendar
- Работает для любой ниши без настройки "под бизнес"
- Шеринг ссылки на расписание в один клик

**Слабые стороны:**
- Нет сбора информации о клиенте — только имя из Telegram
- Нет поля "описание проблемы" — после брони нужно отдельно уточнять
- Нет уведомлений в Telegram владельцу (только email)
- Нет истории клиентов и базы данных
- Не подходит для работы с незнакомыми клиентами (ориентирован на b2b/встречи)
- Нет управления статусами заявок

**Что мы можем сделать лучше:**
- Добавить форму с контекстом: телефон + описание проблемы
- Уведомления владельцу прямо в Telegram с полной карточкой клиента
- База всех клиентов в Google Sheets с историей обращений

### 3. FitWithAnna (фитнес-коучинг Mini App)

**Что делает:** Telegram Mini App персонального тренера: клиент подписывается, получает план тренировок, может записаться на персональное занятие, следить за прогрессом. Оплата через Telegram Payments (Stars или карта).

**Сильные стороны:**
- Полноценный продукт внутри одного мессенджера
- Подписочная модель — стабильный доход
- Видеоконтент и план прямо в приложении
- 95% удержание клиентов после 3 месяцев
- Красивые карточки тренировок, прогресс-бар
- Запись на живые сессии в 2 клика

**Слабые стороны:**
- Высокая стоимость разработки (полноценный продукт)
- Нет offline-режима — всё зависит от интернета
- Сложно масштабировать на нишевые сервисы без переработки
- Интерфейс заточен под визуальный контент, не под текстовые заявки

**Что мы можем сделать лучше:**
- Взять идею "красивых карточек" для отображения типов услуг (натяжка, ремонт рамы, замена ручки)
- Прогресс диалога при записи ("Шаг 2 из 4") — клиент видит, сколько осталось

### 4. Pulse Events (билеты и управление мероприятиями)

**Что делает:** Mini App для продажи билетов на события, QR-check-in на входе, управление гостевым списком. Организатор видит всех зарегистрированных, может ставить статусы.

**Сильные стороны:**
- Скорость регистрации: +300% (с 3 до 10+ человек в минуту на входе)
- Уникальные QR-коды для каждого клиента
- Панель организатора прямо в Telegram
- Push-напоминания о мероприятии
- История посещений клиента

**Слабые стороны:**
- Очень нишевое решение (только события)
- Нет сбора "свободного" описания от клиента
- Управление статусами только бинарное: "пришёл / не пришёл"

**Что мы можем сделать лучше:**
- Взять идею статусов заявки: "Новая → Принята → Выполнена"
- Уведомления клиенту при смене статуса ("Ваша ракетка готова!")

### 5. Dental/Medical Booking Mini Apps (стоматология, клиники)

**Что делает:** Несколько независимых решений для медицинских клиник — пациент видит свободные окна врачей, выбирает время, вносит предоплату, может перенести визит. Особенность: нужно собирать медицинскую информацию (симптомы, анамнез).

**Сильные стороны:**
- Поле описания симптомов — ключевое для понимания задачи ДО визита
- Предоплата снижает количество no-show на 60-70%
- Перенос визита самим клиентом без звонка врачу
- Карточка пациента с историей визитов
- Напоминания за 24ч и 1ч до приёма

**Слабые стороны:**
- Тяжёлые формы — клиенты бросают на полпути (4+ поля подряд)
- Переизбыток информации на экране
- Нет чата с врачом внутри Mini App
- Плохая мобильная адаптация у большинства решений

**Что мы можем сделать лучше:**
- Короткий диалог: не форма, а пошаговые вопросы (как в нашем боте)
- Не более 1 вопроса на экран / на шаг диалога
- Описание проблемы — обязательное поле, как у медиков

## Обязательные UI-элементы

### Telegram Native компоненты (использовать обязательно)
- **MainButton** — главный CTA на каждом шаге ("Далее", "Подтвердить", "Записаться")
- **BackButton** — нативная кнопка назад Telegram (не делать свою)
- **themeParams** — цвета из темы пользователя (тёмная / светлая автоматически)
- **HapticFeedback** — вибрация при успехе/ошибке (ощущение "нативного" приложения)
- **showPopup / showAlert** — системные диалоги подтверждения
- **ClosingConfirmation** — защита от случайного закрытия при заполнении формы

### UI-паттерны
- Прогресс-индикатор шагов ("Шаг 2 из 4")
- Skeleton-экраны при загрузке (не пустой белый экран)
- Один вопрос / одно действие на экране
- Аватар и имя пользователя из Telegram — не просить заново
- Кнопки с закруглёнными углами, крупные (44px+) — мобильный тач
- Inline-клавиатура или Quick Reply для выбора из готовых вариантов

## Что мы можем сделать лучше конкурентов

### 1. Контекстная форма вместо холодного календаря
Конкуренты показывают пустой календарь без понимания задачи клиента. Мы сначала узнаём **что сломалось** — и только потом предлагаем время. Мастер приходит подготовленным.

### 2. Описание проблемы как обязательный шаг
Ни один из рассмотренных сервисов не делает текстовое описание **обязательным**. Для ремонтной ниши это критично — мастер должен знать заранее, нужны ли запчасти.

### 3. Статусы заявки для клиента
Клиент не в вакууме: "Заявка принята → Ждём ракетку → Ремонт начат → Готово к выдаче". Автоматические уведомления в Telegram.

### 4. Нативный Telegram-дизайн
Большинство Mini Apps — это сайты в iframe. Мы делаем приложение, которое выглядит как часть Telegram: цвета из темы, нативные кнопки, haptic feedback.

### 5. Работает без регистрации
Ноль паролей, ноль форм "создать аккаунт". Имя из Telegram уже есть — просим только телефон и описание.

### 6. Google Sheets как CRM для старта
Простой старт без SaaS-подписок. Вся база клиентов сразу видна в таблице, можно фильтровать, ставить статусы, добавлять комментарии.

## Рекомендуемый flow записи (архитектура экранов)

```
Приветствие / онбординг
        ↓
Выбор типа услуги (карточки: натяжка / ремонт рамы / замена ручки / диагностика)
        ↓
Описание проблемы (текст + фото опционально)
        ↓
Выбор удобной даты
        ↓
Выбор времени (доступные слоты)
        ↓
Контактные данные (имя + телефон, имя подтянуть из Telegram)
        ↓
Экран подтверждения (резюме заявки)
        ↓
Экран успеха (номер заявки + что будет дальше)
        ↓
Мои заявки (история + статусы)
```

## Топ-15 болей пользователей в booking-приложениях

1. **Форма-кирпич** — 4+ поля подряд, клиенты бросают
2. **Скрытая цена** — стоимость появляется только на шаге оплаты
3. **Нет описания проблемы** — после записи приходится уточнять всё заново
4. **Слоты заняты, но нет объяснения** — непонятно когда вообще можно
5. **Нет отмены/переноса** — клиент звонит мастеру вручную
6. **Generic ошибки** — "Что-то пошло не так" без подсказки что делать
7. **Нет статуса заявки** — клиент не знает, приняли его или нет
8. **Тормозящий календарь** — долгая отрисовка, лаги на слабых устройствах
9. **Нет подтверждения на экране успеха** — куда смотреть потом?
10. **Клавиатура перекрывает форму** — поле ввода уходит за экран
11. **Нет напоминаний** — клиент забывает про запись
12. **Выбор мастера без описания** — нет фото, опыта, специализации
13. **Зависание при смене интернета** — нет оффлайн-fallback
14. **Нет истории** — клиент не помнит что и когда делал
15. **Нет обратной связи** — не понятно, нажалась кнопка или нет

## Экспертная оценка

---

### 👁 UX-дизайнер

#### Типичные ошибки новичков

**1. Форма вместо диалога**
Новички делают одну длинную форму: "Имя, Телефон, Услуга, Дата, Описание" — и клиент её не заполняет. Правильно: один шаг = один вопрос. Это не каприз — это психология завершённости. Каждый следующий шаг кажется маленьким.

**2. Свой UI вместо Telegram-нативного**
Рисуют свои кнопки "Далее", свою кнопку "Назад", своё модальное окно подтверждения. Всё это ломает ощущение нативности. Пользователь Telegram ожидает поведения Telegram — используйте MainButton, BackButton, showPopup.

**3. Не показывают прогресс**
Клиент не знает, сколько шагов осталось. После второго экрана он думает: "А сколько ещё?" и закрывает. Решение: "Шаг 2 из 4" — простая строка сверху.

**4. Не подтягивают данные из Telegram**
Спрашивают имя у того, чьё имя уже есть в `WebAppUser`. Клиент видит пустое поле там, где могло быть его имя — первое впечатление испорчено.

**5. Экран успеха — пустышка**
После подтверждения — просто "Спасибо". Клиент не знает: когда перезвонят? Что делать дальше? Куда смотреть? Хороший success-screen: номер заявки, ожидаемый ответ ("Мы свяжемся в течение часа"), кнопка "Мои заявки".

**6. Не закрывают форму при случайном свайпе**
Клиент заполнил 3 поля, случайно смахнул — Mini App закрылся, данные потеряны. Подключайте `enableClosingConfirmation()` на шагах с вводом.

**7. Кнопки 32px на мобильном**
Минимум тач-таргета — 44×44px. Маленькие кнопки раздражают и вызывают ошибки нажатия, особенно при выборе даты и времени.

#### Что обязательно должно быть

- **Прогресс-бар или "Шаг N из M"** — на каждом шаге многоэтапного флоу
- **Имя подтянуто из Telegram автоматически** — не заставлять вводить
- **Inline-варианты ответа** — где возможно, кнопки выбора вместо текстового поля
- **Skeleton при загрузке** — никаких белых экранов
- **Подтверждение-резюме** — перед финальной отправкой показать всё одним экраном
- **Success-экран с деталями** — номер заявки + что будет дальше
- **ClosingConfirmation** на шагах с вводом данных

#### Что бесит пользователей сильнее всего

1. Спросили телефон — потом спросили снова на другом шаге
2. Нажал "Назад" — вернулся на самый начало, не на предыдущий шаг
3. Ввёл описание 5 предложениями — форма не отправилась из-за "ошибки" без объяснения
4. Выбрал дату — слот оказался занят, но сообщение об этом появилось после подтверждения
5. Получил "Ваша заявка принята" — и тишина на 2 дня. Куда звонить?

### 📣 Маркетолог

#### Что заставляет записаться (триггеры конверсии)

**Доверие — главный барьер**
Человек открывает незнакомый Mini App. Первые 3 секунды — решение: закрыть или остаться. Нужно немедленно снять тревогу:
- Фото мастера (реальное, не стоковое)
- Имя и стаж: "Алексей, 8 лет практики"
- Конкретное обещание: "Ответим в течение 1 часа"

**Социальное доказательство**
- Счётчик: "Уже 230 ракеток отремонтировано"
- Отзывы прямо на стартовом экране — 2-3 коротких, реальных
- "Последний заказ: 20 минут назад" — живость бизнеса

**Снижение порога входа**
- Первый шаг должен быть ненапряжным: "Выберите тип проблемы" (кнопки, не текст)
- Не спрашивать телефон сразу — сначала узнать о ракетке, потом контакты
- Фраза "Запись бесплатна и ни к чему не обязывает" снимает страх обязательства

**Срочность и ограниченность**
- "Ближайшее свободное время: завтра 15:00" — показывать конкретный слот, а не абстрактный календарь
- "Осталось 2 свободных слота на этой неделе" — дефицит увеличивает конверсию

**Ценность до записи**
- Прайс-лист с ориентировочными ценами: "Натяжка от 500 ₽ — готово за 1 день"
- Это убирает главный вопрос "а сколько стоит?" и отсеивает нецелевых

#### Элементы, повышающие конверсию (приоритет)

| Элемент | Эффект | Приоритет |
|---------|--------|-----------|
| Фото мастера на стартовом экране | +доверие | 🔴 Критично |
| Счётчик выполненных работ | +доверие | 🔴 Критично |
| Ориентировочные цены / прайс | -тревога о деньгах | 🔴 Критично |
| "Ответим в течение 1 часа" | -неопределённость | 🟠 Важно |
| 2-3 коротких отзыва | +соцдоказательство | 🟠 Важно |
| "Ближайший слот: завтра 15:00" | +срочность | 🟠 Важно |
| Прогресс-бар ("Шаг 2 из 4") | -тревога о длине | 🟡 Желательно |
| Анимация успеха при отправке | +удовольствие | 🟡 Желательно |

#### Что убивает конверсию

- Просят телефон на первом же экране (слишком рано — человек ещё не доверяет)
- Нет цен: "уточняйте при записи" — это сигнал о скрытых расходах
- Долгая загрузка — каждые 100ms задержки снижают конверсию на ~1%
- Нет подтверждения после записи — клиент не уверен, что всё прошло
- Нет ответа дольше 2 часов после заявки — холодный лид становится мёртвым

#### Стратегия первого касания

Самый эффективный путь клиента в нишевом Telegram-боте:
```
Клиент видит бота в чате / находит по рекомендации
    → Первый экран: "Привет, я Алексей. Ремонтирую ракетки 8 лет.
                     230 клиентов. Начнём?" [КНОПКА]
    → Выбор типа проблемы (3-4 кнопки — минимальное усилие)
    → Описание (уже вовлечён)
    → Дата/время
    → Только здесь — телефон (уже доверяет, уже вложился)
    → Успех + ожидание
```
Принцип: сначала вовлечь, потом просить контакт.

### ⚙️ Telegram-разработчик

#### Жёсткие ограничения, о которых нужно знать

**Viewport и клавиатура**
Mini App запускается в webview с ограниченной высотой. При открытии клавиатуры viewport сжимается — если не обрабатывать `viewportChanged`, поле ввода уходит под клавиатуру. Обязательно:
```js
Telegram.WebApp.onEvent('viewportChanged', () => {
  // пересчитать layout
});
```

**iOS swipe-конфликт**
Горизонтальный свайп от края экрана — системный жест iOS "закрыть приложение". Если у вас есть горизонтальный скролл (карточки, карусель) — пользователи будут случайно закрывать Mini App. Либо отказаться от горизонтального скролла, либо добавлять padding от краёв ≥ 16px.

**Нет фоновой работы**
Mini App полностью выгружается при закрытии. Нельзя хранить состояние в памяти между сессиями — используйте `localStorage` или backend.

**Telegram Payments — комиссия и ограничения**
Stars — только для цифровых товаров, не для физических услуг. Для оплаты ремонта нужен внешний провайдер (ЮKassa, Stripe через провайдера). Telegram берёт свою комиссию плюс провайдер. Учитывайте при решении о предоплате.

**CSP и внешние ресурсы**
Telegram ограничивает Content Security Policy. Если грузите шрифты или иконки с CDN — могут блокироваться. Лучше бандлить всё локально или использовать разрешённые домены.

**Размер бандла**
Нет жёсткого лимита, но Mini App должен загружаться за ≤ 2.5 секунды (LCP). Большой React-бандл убьёт этот показатель. Используйте code splitting, lazy loading изображений, минимум зависимостей.

#### Что работает хорошо

- **MainButton** — лучшее, что есть в SDK. Нативный, всегда виден, не требует места в layout. Используйте агрессивно.
- **HapticFeedback** — добавляет 0 строк в layout, но ощутимо улучшает "чувство" приложения. Обязательно на подтверждение и ошибку.
- **themeParams** — за 5 строк CSS Variables получаете полную адаптацию к тёмной/светлой теме Telegram. Не игнорируйте.
- **WebAppUser** — бесплатные данные пользователя без регистрации. Имя, фамилия, username, language_code. Используйте.
- **showPopup / showConfirm** — нативные диалоги выглядят как системные, вызывают больше доверия чем кастомные модалки.

#### Что лучше не делать

| Антипаттерн | Почему плохо | Замена |
|-------------|-------------|--------|
| Своя кнопка "Назад" | Конфликт с нативным BackButton, двойная навигация | Только `BackButton` из SDK |
| Кастомное модальное окно подтверждения | Выглядит как фишинг, пользователь не доверяет | `showConfirm()` из SDK |
| Горизонтальный скролл у края экрана | Конфликт со свайпом iOS | Padding ≥ 16px или отказаться |
| `alert()` / `confirm()` браузерные | Могут не работать в webview Telegram | `showPopup()` из SDK |
| Хранить состояние только в памяти | Теряется при закрытии | `localStorage` + backend |
| Большой React без SSR | Долгая загрузка, белый экран | Preact / Vite / skeleton |
| Внешние шрифты через @import | Могут блокироваться CSP | Бандлить локально |
| Запрашивать геолокацию без объяснения | Пользователь отказывает | Объяснить зачем, потом запросить |

#### Архитектурные решения для этого проекта

**Фронтенд:** React + Vite (быстрая сборка) или Preact (меньше бандл). Telegram UI Kit (`@telegram-apps/telegram-ui`) — готовые нативные компоненты.

**Бэкенд:** Python (уже начат) + FastAPI для webhook от бота. Mini App делает запросы на тот же бэкенд.

**Данные:** Google Sheets через gspread — уже реализовано. Для Mini App добавить REST endpoint, который возвращает заявки клиента по его Telegram ID.

**Аутентификация:** Проверять `initData` от Telegram на бэкенде — это единственный надёжный способ знать, кто открыл Mini App. Не хранить сессии.

**Деплой:** Telegram требует HTTPS. Vercel/Railway/Render — бесплатный старт с SSL из коробки.
